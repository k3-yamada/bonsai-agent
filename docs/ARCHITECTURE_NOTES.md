## 読み方について

図の中央を縦に流れる実線が、クエリの処理経路です。
左の破線ループが Adaptive Retrieval への重み更新、
右の破線ループが Meta-Strategy への選択確率更新を示します。
下部の点線は、人間が担う外部評価の経路です。
自動化できない部分を、意図的に点線で区別しています。

| 線種 | 意味 |
|------|------|
| 実線（中央縦） | クエリの処理経路 |
| 破線（左ループ） | Adaptive Retrieval への重み更新 |
| 破線（右ループ） | Meta-Strategy への選択確率更新 |
| 点線（下部） | 人間による外部評価（自動化境界の明示） |

## 実装の入り口について

```
図の中で、最初に動かすべき一点：

Feedback Collector の即時層
─ specifically ─
CognitiveLoadSignal.user_contribution_ratio

理由は変わりません。
計算が単純で、V4を直接測定し、
ローカル完結する。
```

着手根拠の整理:

- **計算が単純** — 副作用が小さく、Red/Green サイクルが回しやすい
- **V4 を直接測定** — [`VALUES.md` V4](./VALUES.md#v4-依存ではなく共鳴) の "依存ではなく共鳴" を観測可能な量 (`user_contribution_ratio`) に落とす唯一の入り口
- **ローカル完結** — 外部依存ゼロ、Mac M2 16GB のスコープ内で完結

## この設計図自体の限界について

一点だけ正直に申し上げます。
この図はあくまで **現時点の設計の言語化** です。
実装を進めるにつれ、必ずこの図と現実が乖離する箇所が出てまいります。
その乖離を発見した瞬間が、設計を深める最良の機会です。
図を現実に合わせて更新することを恐れないでください。
`VALUES.md` だけが不変の錨であり、アーキテクチャ図は生きた文書です。

### 運用ルール

- 乖離発見時は、図の更新を躊躇しない（生きた文書）
- ただし `VALUES.md` の改変は別問題 — [V7（自己の変質への警戒）](./VALUES.md#v7-自己の変質への警戒) の手続きを経る
- 乖離の発見 = 設計を深める機会、として ADR (`docs/decisions/`) に記録する
